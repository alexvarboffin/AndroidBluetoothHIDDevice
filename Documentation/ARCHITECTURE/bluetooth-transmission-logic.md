# Анализ способов передачи текстовых данных по Bluetooth

В ходе разработки проекта AndroidBluetoothHIDDevice был проведен сравнительный анализ двух основных подходов к передаче ввода:

## 1. HID (Human Interface Device) — Текущий метод
**Принцип:** Смартфон эмулирует физическую клавиатуру.
- **Что передается:** Скан-коды (Scan Codes) — идентификаторы физических клавиш (например, "кнопка №4").
- **Зависимость:** Результат (символ) полностью зависит от текущей раскладки (Layout) на стороне принимающего устройства (ПК).
- **Плюсы:** Не требует установки ПО на ПК; работает везде (BIOS, Smart TV, iOS/Android).
- **Минусы:** Невозможно передать конкретный символ (например, "A") независимо от раскладки хоста.

## 2. SPP (Serial Port Profile) / BLE GATT — Метод "Сокета"
**Принцип:** Создание прозрачного канала передачи данных между двумя приложениями.
- **Что передается:** Готовые данные (Unicode строки, байты, JSON).
- **Зависимость:** Не зависит от раскладки. Передавая `U+041F` ("П"), хост получает именно этот символ.
- **Плюсы:** Полная независимость от настроек системы; высокая скорость и надежность передачи текста.
- **Минусы:** **Обязательно наличие программы-приемника** на стороне ПК, которая будет считывать данные из порта и эмулировать ввод в ОС.

## Резюме для проекта
Для сохранения концепции **"универсальной клавиатуры без драйверов"** был выбран метод **HID**. Ограничение с раскладками признано фундаментальной особенностью протокола.

### Решения проблемы раскладки в рамках HID:
1. **Эмуляция Alt-кодов (Windows):** Передача символа через быстрый набор `Alt + [код]`. (Сложно, медленно, только для Win).
2. **Дистанционное переключение:** Добавление кнопок эмуляции системных сочетаний (`Alt+Shift`, `Cmd+Space`) для смены языка на хосте прямо из приложения.
